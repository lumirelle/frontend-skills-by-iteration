# UI Reading Guide

读 `docs/vX.Y.Z/ui/*` 设计稿时用的通用方法。目标：把图里**确定**的信息提取成需求/方案输入，把**不确定**的信息显式标为假设或 open question——不靠猜像素。

适用：步骤 1（`frontend-requirements`）提取需求，步骤 2（`frontend-design`）确认组件拆分与交互。

## 提取维度

逐张图按以下维度看，只记图上**能确认**的事实：

| 维度 | 看什么 | 产出去向 |
|------|--------|----------|
| 布局结构 | 整体分区（头/内容/底/侧栏）、主次区域、栅格/分栏 | summarized「页面与模块」；design「组件结构」 |
| 组件清单 | 可识别控件：按钮、输入、下拉、表格、卡片、Tab、弹窗、列表项 | summarized「页面与模块」；design「组件结构」 |
| 层级与重复 | 哪些是重复单元（列表项/卡片）、哪些是容器 | design 复用 vs 新建判断 |
| 交互状态 | 同一元素的多态：默认/hover/选中/禁用/加载/空/错误 | summarized「状态与交互」 |
| 文案 | 标题、标签、占位符、按钮文字、提示语、空态文案 | summarized 文案；design 直接引用 |
| 数据字段 | 展示的字段、格式（日期/金额/单位）、条件展示 | summarized「数据与展示」 |
| 层级动作 | 主操作 vs 次操作、危险操作、跳转入口 | summarized「用户流程」 |
| 响应式线索 | 是否给了多尺寸稿、断点差异、移动端变化 | summarized「平台差异」 |

## 多状态稿的读法

设计稿常把一个页面的多个状态画成并排的多张图或多个画板：

- 同名带状态后缀（如 `order-list-empty.*`、`order-list-error.*`）→ 归到同一页的不同状态，写进「状态与交互」表，不拆成不同页面。
- 一张图里并排画了多态 → 在 summarized 注明「来自同一张稿的 N 个状态」。
- 稿里**没画**的状态（如 loading、无权限）→ 不要假装看到；按需求补为假设或 open question。

## 确定 vs 不确定

| 情况 | 处理 |
|------|------|
| 图上明确画出 | 作为事实写入对应章节 |
| 图上有但语义不明（图标无标签、占位文字 lorem） | 写「假设」，并说明你的解读 |
| 图上没有但功能需要（边界态、错误态） | 写 open question 或假设，不臆造视觉细节 |
| 图与 origin PRD 冲突 | **不裁决**，列入 open questions 让用户确认（UI 优先仅在无冲突时适用） |
| 精确像素/间距/色值 | 不写死在需求里；交由实现阶段对照稿与设计系统 |

## 反模式

- 编造稿里看不清的文案或字段名。
- 把一张多状态稿拆成多个「页面」需求。
- 用「大概」「应该是」描述交互却不标假设。
- 纠结像素级数值，写进验收标准（应是行为/可见结果级）。
- 无 UI 稿时凭空补充视觉细节而不标注。

## 无 UI 稿

- summarized 头部标注「无 UI 稿」（模板 `UI: 无`）。
- 布局与视觉细节写为假设或 open question。
- 交互行为仍按 origin + 项目既有同类页面推断，并标注来源。
