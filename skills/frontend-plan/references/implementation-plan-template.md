# Implementation Plan Template

复制此结构生成 `docs/vX.Y.Z/plans/<name>.md`。删除不适用的章节，不留空占位。

```markdown
# [页面/功能名] 实施计划

> 版本：vX.Y.Z | 来源：design/<name>.md

## 目标

一句话说明本计划交付什么。

## 范围

**包含：**

- …

**不包含：**

- …

## 前置确认

| 项 | 状态 | 说明 |
|----|------|------|
| summarized 已确认 | 是/否 | |
| design 已确认 | 是/否 | |
| open questions 已关闭 | 是/否 | |

未确认项必须在进入代码实现前处理。

## 文件边界

| 操作 | 文件 | 说明 |
|------|------|------|
| 新增 | `src/...` | |
| 修改 | `src/...` | |
| 测试 | `src/...` / `tests/...` | |

## 任务列表

### Task 1: [任务名]

**目标**：…

**依赖**：无

- 修改：`src/...`

1. …

**验证：**

- 运行：`<test command>`
- 通过条件：…

## Open Questions

- …
```

## 填写要点

| 章节 | 要求 |
|------|------|
| 范围 | 与 design 对齐，避免实现阶段扩 scope |
| 文件边界 | 精确到文件路径；未知则先探查，不猜 |
| 任务列表 | 每个任务可独立完成和验证 |
| 步骤 | 写动作，不写大段解释；避免“实现功能”这种空步骤 |
| 验证 | 尽量写具体命令；未知命令从 technical-architecture 获取 |
| 测试矩阵 | 每条验收标准至少有一个测试维度和覆盖方式 |
| Open Questions | 有阻塞问题时不得进入代码实现 |
